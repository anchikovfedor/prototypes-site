# Справочник стандартов анимации

Точные значения, кривые и правила, на которые опирается ревью. В findings цитировать их,
а не приближать. Дистиллят философии дизайн-инжиниринга Эмиля Ковальски
([animations.dev](https://animations.dev/)).

## Анимировать ли вообще? (таблица частоты)

| Частота | Решение |
| --- | --- |
| 100+ раз/день (клавиатурные шорткаты, тоггл command palette) | Без анимации. Совсем. |
| Десятки раз/день (hover-эффекты, навигация по спискам) | Убрать или радикально сократить |
| Изредка (модалки, шторки, тосты) | Стандартная анимация |
| Редко / первый раз (онбординг, фидбек, празднования) | Допустим delight |

**Никогда не анимировать действия с клавиатуры** — они повторяются сотни раз в день;
анимация делает их медленными и «отвязанными». (У Raycast нет анимации открытия/закрытия —
верно для того, чем пользуются сотни раз в день.)

Валидные цели motion: пространственная связность, индикация состояния, объяснение,
фидбек, сглаживание резкой смены. «Красиво выглядит» на часто видимом элементе — не цель.

## Easing

Порядок выбора:
- Появление или исчезновение → **`ease-out`** (стартует быстро, ощущается отзывчиво)
- Движение / морф по экрану → **`ease-in-out`**
- Hover / смена цвета → **`ease`**
- Постоянное движение (marquee, прогресс) → **`linear`**
- По умолчанию → **`ease-out`**

**Никогда `ease-in` на UI.** Он стартует медленно, затягивая именно тот момент, на который
смотрит пользователь. `ease-out` за 200 мс *ощущается* быстрее, чем `ease-in` за 200 мс.

Встроенные CSS-easing слабые. Использовать сильные кастомные кривые:

```css
--ease-out: cubic-bezier(0.23, 1, 0.32, 1);        /* сильный ease-out для UI */
--ease-in-out: cubic-bezier(0.77, 0, 0.175, 1);    /* сильный ease-in-out для движения по экрану */
--ease-drawer: cubic-bezier(0.32, 0.72, 0, 1);     /* iOS-подобная кривая шторки (Ionic) */
```

Кривые брать на [easing.dev](https://easing.dev/) или [easings.co](https://easings.co/) —
не изобретать с нуля.

## Длительность

| Диапазон | Значение | Применение |
| --- | --- | --- |
| Short | 100–200 мс | Иконки, чекбоксы, свичи, `:hover` и `:active` |
| Medium | 250–400 мс | Модальные окна, снекбары, дропдауны, навигация |
| Long | 450–600 мс | Крупные шторки, переходы между экранами |
| Extra long | 700–1000 мс | Онбординги, иллюстративные и промо-анимации |

**Сетка 50 мс.** Все длительности и задержки (включая stagger) кратны 50 мс: 100, 150,
200… Значение вне сетки (например 160 мс или 180 мс) — finding.

**Правило: держаться нижней части диапазона своего класса.** Мелкий и частый UI — до
300 мс; крупные шторки и переходы между экранами — до 600 мс; 700–1000 мс — только
иллюстративное/промо, не интерактивный UI. Дропдаун за 150 мс ощущается отзывчивее, чем
за 400 мс. Быстрый спиннер делает загрузку субъективно быстрее (реальное время то же).
Мгновенные тултипы после первого (без задержки и анимации) делают тулбар быстрее.

## Физичность

- **Никогда `scale(0)`.** Старт от `scale(0.9–0.97)` + `opacity: 0`. В реальном мире ничто
  не появляется из ничего.
- **Поповеры с учётом origin.** Масштаб от триггера, не от центра:
  ```css
  .popover { transform-origin: var(--radix-popover-content-transform-origin); } /* Radix */
  .popover { transform-origin: var(--transform-origin); }                       /* Base UI */
  ```
  **Модалки — исключение**: появляются по центру вьюпорта, `transform-origin: center`
  остаётся.
- **Фидбек нажатия кнопки.** `transform: scale(0.97)` на `:active`,
  `transition: transform 150ms ease-out`. Деликатно (0.95–0.98). Для любого нажимаемого
  элемента.

## Пружины

Ощущаются естественно, потому что симулируют физику; фиксированной длительности нет —
затухание определяют параметры. Использовать для: drag с инерцией, «живых» элементов
(Dynamic Island), прерываемых жестов, декоративного слежения за курсором.

```js
// Apple-стиль (проще рассуждать) — рекомендуется
{ type: "spring", duration: 0.5, bounce: 0.2 }

// Классическая физика (больше контроля)
{ type: "spring", mass: 1, stiffness: 100, damping: 10 }
```

Bounce держать деликатным (0.1–0.3); в большинстве UI избегать — оставить для
drag-to-dismiss и игривых взаимодействий. Пружины сохраняют скорость при прерывании
(keyframes рестартуют с нуля) — идеальны для жестов, которые пользователь может развернуть
на полпути.

Взаимодействия с мышью: интерполировать через `useSpring`, а не привязывать значение к
позиции мыши напрямую (напрямую = искусственно, без инерции). Только для декоративного
motion.

## Прерываемость

CSS **transitions** можно прервать и ретаргетить на середине; **keyframes** рестартуют с
нуля. Для всего, что триггерится часто (добавление тостов, тогглы), transitions плавнее.

```css
/* Прерываемо — хорошо для динамического UI */
.toast { transition: transform 400ms ease; }

/* Непрерываемо — избегать для динамического UI */
@keyframes slideIn { from { transform: translateY(100%); } to { transform: translateY(0); } }
```

`@starting-style` — появление без JS:

```css
.toast {
  opacity: 1; transform: translateY(0);
  transition: opacity 400ms ease, transform 400ms ease;
  @starting-style { opacity: 0; transform: translateY(100%); }
}
```

Legacy-фолбэк: `useEffect(() => setMounted(true), [])` + атрибут `data-mounted`.

## Асимметричный тайминг

Медленно там, где пользователь решает; быстро там, где отвечает система.

```css
.overlay { transition: clip-path 200ms ease-out; }            /* отпускание: быстро */
.button:active .overlay { transition: clip-path 2s linear; }  /* нажатие: медленно, осознанно */
```

## Перфоманс

- **Анимировать только `transform` и `opacity`** — они минуют layout/paint и идут на GPU.
  `padding`/`margin`/`height`/`width`/`top`/`left` запускают все три шага рендера.
- **Не гонять transform детей через CSS-переменную на родителе** — пересчёт стилей у всех
  детей. Ставить `transform` прямо на элемент.
  ```js
  element.style.setProperty('--swipe-amount', `${d}px`); // плохо: recalc у всех детей
  element.style.transform = `translateY(${d}px)`;        // хорошо: только этот элемент
  ```
- **Шортхенды Framer Motion НЕ аппаратно-ускорены.** `x`/`y`/`scale` идут в main thread
  через rAF и роняют кадры под нагрузкой. Использовать полную transform-строку:
  ```jsx
  <motion.div animate={{ x: 100 }} />                          // роняет кадры под нагрузкой
  <motion.div animate={{ transform: "translateX(100px)" }} />  // аппаратно ускорено
  ```
- **CSS-анимации выигрывают у JS под нагрузкой** — работают вне main thread; rAF-анимации
  заикаются, пока браузер грузит/скриптует/рисует. CSS — для предопределённого motion,
  JS — для динамики/прерываемости.
- **WAAPI** — JS-контроль с производительностью CSS (аппаратное ускорение, прерываемость,
  без библиотек):
  ```js
  element.animate([{ clipPath: 'inset(0 0 100% 0)' }, { clipPath: 'inset(0 0 0 0)' }],
    { duration: 1000, fill: 'forwards', easing: 'cubic-bezier(0.77, 0, 0.175, 1)' });
  ```

## Transforms и clip-path

- **Проценты в `translate`** — от размера самого элемента: `translateY(100%)` сдвигает на
  его высоту при любых габаритах (так Sonner/Vaul позиционируют тосты/шторки).
  Предпочитать захардкоженным px.
- **`scale()` масштабирует и детей** (шрифт, иконки, контент) — фича для фидбека нажатия.
- **3D**: `rotateX/Y` + `transform-style: preserve-3d` — глубина/орбита/флип без JS.
- **`clip-path: inset(t r b l)`** — мощный анимационный инструмент: каждое значение
  «съедает» со своей стороны. Применения: reveal-on-scroll (`inset(0 0 100% 0)` →
  `inset(0 0 0 0)`), оверлей hold-to-delete, бесшовные цветовые переходы табов
  (дубликат + клип активной копии), слайдеры сравнения.

## Жесты и drag

- **Дисмисс по инерции**: не требовать пересечения дистанции — считать скорость
  (`Math.abs(distance)/elapsedMs`); дисмисс при `> ~0.11`. Флика должно хватать.
- **Демпфирование на границах**: перетаскивание за естественный край двигает всё меньше
  (реальные вещи замедляются перед остановкой).
- **Pointer capture** после старта драга — жест продолжается, когда курсор вышел за
  границы.
- **Защита от мультитача**: игнорировать лишние касания после старта драга
  (`if (isDragging) return`) — предотвращает скачки.
- **Трение вместо жёстких стопов** — допускать over-drag с нарастающим сопротивлением, а
  не невидимую стену.

## Маскировка неидеальных кроссфейдов

Если кроссфейд показывает два наложенных состояния несмотря на подгонку
easing/длительности — добавить лёгкий `filter: blur(2px)` на время перехода, чтобы слить
их в одно воспринимаемое превращение. Blur < 20px (тяжёлый blur дорог, особенно в Safari).

## Stagger

Групповые появления стаггерить; шаг — 50 мс (по сетке 50 мс; 100 мс и дольше между
элементами — ощущается медленно). Stagger декоративен — никогда не блокировать
взаимодействие, пока он играет.

```css
.item { opacity: 0; transform: translateY(8px); animation: fadeIn 300ms ease-out forwards; }
.item:nth-child(2) { animation-delay: 50ms; }
.item:nth-child(3) { animation-delay: 100ms; }
@keyframes fadeIn { to { opacity: 1; transform: translateY(0); } }
```

## Доступность

```css
@media (prefers-reduced-motion: reduce) {
  .element { animation: fade 0.2s ease; } /* opacity/цвет остаются, transform-движение убирается */
}
@media (hover: hover) and (pointer: fine) {
  .element:hover { transform: scale(1.05); } /* гейт hover-motion — тач даёт ложный hover по тапу */
}
```

```jsx
const reduce = useReducedMotion();
const closedX = reduce ? 0 : '-100%';
```

Reduced motion = меньше и мягче анимаций, не ноль: переходы, помогающие пониманию,
остаются; движение/смена позиций убираются.

## Отладка (рекомендовать в ревью, когда ощущение под вопросом)

- **Slow motion**: длительность ×2–5 или инспектор анимаций в DevTools. Проверить: цвета
  кроссфейдятся чисто, easing не обрывается резко, `transform-origin` верный,
  согласованные свойства синхронны.
- **Покадрово**: панель Animations в Chrome DevTools вскрывает дрейф тайминга между
  согласованными свойствами.
- **Реальные устройства** для жестов (шторки, свайпы) — телефон в сети, dev-сервер по IP,
  remote-devtools Safari.
- **Свежий взгляд назавтра** — огрехи, невидимые во время разработки, всплывают позже.

## Целостность

Motion — под характер компонента: игривое может пружинить сильнее; профессиональный
дашборд — чёткий и быстрый. Sonner ощущается правильным отчасти потому, что easing,
длительность, дизайн и даже имя — в гармонии: чуть медленнее, `ease` вместо `ease-out`,
ради элегантности. Opacity + height в появляющихся/исчезающих списках — метод проб;
формулы нет — крутить, пока не почувствуется правильно.
